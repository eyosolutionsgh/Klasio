# 06 — Engineering Practices & Tooling

## 6.1 Repository & git hygiene

- Monorepo: pnpm workspaces + Turborepo task caching.
- **Husky** git hooks:
  - `pre-commit`: lint-staged → ESLint + Prettier on staged files; type-check affected packages; secret scan (gitleaks).
  - `commit-msg`: commitlint (Conventional Commits) — enables automated changelogs and semantic releases.
  - `pre-push`: unit tests for affected packages; **migration integrity check** (see 6.3).
- Branching: trunk-based with short-lived feature branches; PR review required; squash merges.
- **No AI trails in git history**: commit messages, PR descriptions, and code comments contain no AI attribution, co-author trailers, or generation markers. Enforced by a commit-msg hook rule (reject `Co-Authored-By: *AI*/Claude/Copilot/ChatGPT`, "Generated with", robot emoji) plus PR-template guidance.

## 6.2 Code quality

- TypeScript `strict` everywhere; ESLint (typescript-eslint + import boundaries plugin to enforce modulith module boundaries); Prettier.
- Testing pyramid: Vitest/Jest unit tests (domain logic: SBA computation, ledger, entitlements are test-first), integration tests against real Postgres (Testcontainers), Playwright E2E for critical flows (invoice→pay→receipt; marks→report card; pickup scan→log).
- Contract tests with recorded fixtures for gateway webhooks (Hubtel/Paystack/Meta) — providers change payloads.
- Definition of done includes: tests, migration check, tooltip/help text for new UI (see 07), audit-log coverage for mutations.

## 6.3 Migration discipline (project requirement)

A fresh environment must reach a complete, working schema with one command.

- Single ordered migration chain per database, committed to the repo; generated by the ORM, reviewed like code.
- **Rules:** never edit an applied migration; roll forward only; no manual SQL in any environment; data migrations are separate, idempotent, and re-runnable; seeds (GES grading schemes, term calendars, report templates, demo data) are versioned code, separate from schema migrations.
- **CI gates on every PR:**
  1. `migrate deploy` onto an empty Postgres → must succeed.
  2. Schema drift check: freshly-migrated schema == ORM schema definition.
  3. Rollback/forward smoke: previous release's DB + new migrations → must succeed (protects standalone boxes that skip versions).
- Standalone updater runs the same `migrate deploy` on upgrade — which is why the chain must always be clean.

## 6.4 CI/CD

- GitHub Actions: lint → typecheck → unit → integration (Testcontainers) → build → migration gates → E2E (on main) → Docker image publish (versioned, multi-arch for LAN mini-PCs).
- Environments: `dev` → `staging` (auto-deploy, seeded demo school) → `prod` (tagged release, manual approve). Standalone channel: `stable` image stream + signed release manifests; boxes update only to signed releases.
- Release management: changesets/semantic-release off Conventional Commits; feature flags (homegrown on the entitlement engine) separate deploy from release.

## 6.5 Monitoring & operations

- **Errors:** Sentry (API, web, mobile, bot) with tenant tagging.
- **Metrics/traces/logs:** OpenTelemetry → Grafana + Loki/Tempo (or a managed equivalent early on).
- **Business-critical alerts:** webhook failure rate, payment-application lag, reconciliation exception count, sync-queue depth per school, WhatsApp template rejection, report-generation failures during report season (traffic is term-driven and spiky — load-test before each term end).
- **Uptime:** external synthetic checks on portal, checkout, bot webhook.
- **LAN-box fleet:** health beacon (version, disk, backup age, sync lag) to a fleet dashboard; alert when a school's box goes silent > N hours; encrypted off-site backup shipping when online.
- **Status page** + in-app incident banner.

## 6.6 Security practices

- OWASP ASVS-guided reviews for payments, auth, child data surfaces; dependency scanning (Renovate + npm audit + Snyk); gitleaks in CI; signed webhooks verified; rate limiting; MFA for school-admin roles; per-tenant data export/erasure workflows (Act 843).
- Principle: the gate scanner and bot never cache another family's data on-device.

## 6.7 Documentation

- ADRs (architecture decision records) in `docs/adr/` — every irreversible choice gets one.
- Runbooks for: gateway outage, reconciliation backlog, LAN box recovery, license reissue, term-end load.
- API reference auto-generated (OpenAPI) for the Advanced-tier public API.
